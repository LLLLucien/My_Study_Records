# 电梯调度算法需求文档

## 1. 需求分析

### 1.1. 项目背景
目前全球电梯市场中，国际知名品牌包括美国的奥的斯、瑞士的迅达、芬兰的通力，以及日本的三菱、日立、东芝等，这些品牌在技术创新和市场份额上占据领先地位。国内方面，既有上海三菱、奥的斯中国、日立中国等合资企业，也有康力电梯、快意电梯、远大智能等本土品牌。全球主流电梯厂商的调度技术均以经典LOOK/SCAN算法为核心底层逻辑，在此之上根据建筑规模、客流特征和使用场景进行了多层级的商业优化。

本项目实现的单电梯LOOK调度算法，正是所有现代商业电梯调度系统的基础原型。项目旨在验证经典电梯调度算法的实际运行效果，使用简化版LOOK算法实现单部10层电梯的调度模拟，支持接收乘客上下楼请求，展示电梯初始状态和最终完整运行路径，保证调度逻辑符合LOOK算法的核心运行规则。

### 1.2. 核心功能需求
- 实现单部10层电梯的运行模拟，无载客数限制
- 支持分阶段批量输入请求：
  1. 先输入外部请求总数量
  2. 再输入电梯初始楼层
  3. 接着批量输入所有外部楼层请求
  4. 最后按顺序输入每个外部请求对应的乘客内部目标楼层
- 采用LOOK调度算法，一次性处理所有外部和内部请求生成最终运行路径
- 初始运行方向根据距离当前楼层最近的请求自动确定
- 外部请求自动排序：同方向请求按楼层顺序排列，先处理距离更近的方向
- 支持重复请求合并：同一楼层的相同类型请求自动去重，不重复记录
- 最终输出纯数字格式的电梯完整停靠序列，用箭头连接体现运行顺序
- 所有请求处理完成后，自动输出最终运行结果并结束程序

## 2. 核心数据结构

### 2.1. 全局辅助数组
- `Red[MAX_FLOOR_COUNT]`：存储高于当前楼层的所有请求（外部 + 内部）
- `blue[MAX_FLOOR_COUNT]`：存储低于当前楼层的所有请求（外部 + 内部）
- `r_i`：Red数组的当前元素个数
- `b_i`：blue数组的当前元素个数

### 2.2. 电梯本体结构体
```c
typedef struct {
    int currentFloor;          // 当前所在楼层（1-10层）
    int externalRequests[10];  // 外部请求队列（最多10个外部请求）
    int internalRequests[10];  // 内部目标请求队列（与外部请求一一对应）
    int externalCount;         // 已输入的外部请求总数量
    int internalCount;         // 已输入的内部目标请求总数量
    int direction;             // 运行方向标志位：0表示向上，1表示向下
    int finalTargetCount;      // 最终目标楼层总数量
    int finalTargets[MAX_TARGET_COUNT]; // 最终目标楼层队列（最多20个目标）
} Elevator;
```

## 3. 核心算法设计

### 3.1. 算法选型
本项目最终选择LOOK扫描优化算法，是综合对比了经典调度算法在单电梯10层场景下的效率、公平性和实际运行合理性后得出的结论。

LOOK算法完美继承了SCAN算法的公平性优势，同时通过"仅响应最远请求即折返"的优化，彻底消除了SCAN算法的无效空驶问题，是单电梯调度场景下综合表现最优的经典算法，与本项目的需求高度匹配。其他算法的核心缺陷如下：

#### 3.1.1. FCFS先来先服务算法
完全按请求时间顺序处理，会导致电梯频繁往返于高低楼层之间，产生大量无效运行距离。例如1楼有人去10楼、2楼有人去1楼的连续请求，会让电梯全程跑满20层，平均等待时间和运行能耗极高，仅适用于极低客流的极端场景，不符合日常电梯使用需求。

#### 3.1.2. SSTF最短寻道优先算法
虽然单次响应速度最快，但存在严重的"饥饿"问题。当电梯中间楼层持续产生新请求时，两端楼层的请求会被无限期推迟。例如电梯在5楼时，若不断有4、5、6楼的请求，10楼的乘客可能永远无法等到电梯，严重违背电梯服务的公平性原则。

#### 3.1.3. SCAN扫描算法
虽解决了SSTF的饥饿问题，但机械地要求电梯必须到达楼层端点（1楼或10楼）才会折返，即使端点没有任何请求。例如电梯在1楼，仅收到2楼和3楼的上行请求时，SCAN算法仍会让电梯空载运行到10楼再折返，产生大量不必要的空驶，运行效率低于LOOK算法。

### 3.2. 算法核心原则
1. 初始方向确定原则：初始方向默认上行；若所有外部请求都低于初始楼层，则初始方向为下行
2. 每次停靠楼层并处理完上下客和新增请求后，重新计算目标队列和运行方向
3. 无任何请求时，电梯保持初始方向确定原则：比较距离当前楼层最近的上行请求和下行请求的距离，距离更近的方向优先；若只有一个方向有请求，则直接选择该方向
4. 同方向请求排序原则：上行方向的请求按楼层升序排列，下行方向的请求按楼层降序排列
5. 方向翻转原则：处理完当前方向的所有请求后，自动翻转方向处理反方向请求
6. 重复请求合并原则：添加新目标时，若已存在相同楼层的请求，则不重复添加
7. 最终路径生成原则：合并所有外部请求和内部请求，按LOOK规则排序后生成完整的最终运行序列停止状态

### 3.3. 系统初始化模块
- 对应函数：`Init_Elevator(Elevator *Elev)`
- 功能：初始化电梯的所有状态字段，为后续请求处理做准备
- 初始化内容：
  - 最终目标楼层数量设为0
  - 最终目标楼层队列所有元素设为-1（表示空）
  - 当前楼层初始化为0（后续由用户输入覆盖）
  - 运行方向初始化为0（上行，后续由初始方向确定函数覆盖）
  - 外部请求数量和内部请求数量均初始化为0

### 3.4. 外部请求输入模块
- 对应函数：`Input_External_Requests(Elevator *Elev)`
- 功能：接收用户输入的外部请求信息，进行合法性校验，并对外部请求进行初步排序
- 输入流程：
  1. 输入外部楼层请求总数量
  2. 输入电梯初始楼层
  3. 批量输入所有外部楼层请求（空格分隔）
- 校验规则：
  - 外部请求数量不能超过最大楼层数（10）
  - 电梯初始楼层不能超过最大楼层数（10）
  - 外部请求楼层不能超过最大楼层数（10）
  - 外部请求楼层不能与电梯初始楼层重复
  - 若任意一项输入非法，提示用户重新输入所有请求
- 后续处理：
  - 将合法的外部请求复制到电梯结构体的外部请求队列
  - 调用`Sort_External_Requests`函数对外部请求进行初步排序

### 3.5. 外部请求排序模块
- 对应函数：`Sort_External_Requests(int *array, int count, int currentFloor, Elevator *Elev)`
- 功能：将外部请求分为高于当前楼层和低于当前楼层两部分，分别进行排序
- 排序逻辑：
  1. 遍历所有外部请求，将高于当前楼层的请求存入Red数组，低于当前楼层的请求存入blue数组
  2. 对Red数组进行升序排序（`Sort_Red_Top`函数）
  3. 对blue数组进行降序排序（`Sort_Blue_Bottom`函数）
  4. 将排序后的Red数组和blue数组合并回外部请求队列

### 3.6. 初始方向确定模块
- 对应函数：`Determine_Initial_Direction(Elevator *Elev)`
- 功能：根据外部请求的分布情况，自动确定电梯的初始运行方向
- 确定逻辑：
  1. 若只有下行请求（Red数组为空），则初始方向为下行（1）
  2. 若只有上行请求（blue数组为空），则初始方向为上行（0）
  3. 若两个方向都有请求，比较最近的上行请求和最近的下行请求与当前楼层的距离
  4. 距离更近的方向优先；若距离相等，优先选择下行方向
  5. 若初始方向为下行，调整外部请求队列的顺序，将blue数组放在前面，Red数组放在后面

### 3.7. 电梯初始信息输出模块
- 对应函数：`Output_ElevatorInfo(Elevator *Elev)`
- 功能：输出电梯的初始状态信息，包括当前楼层、目标楼层数量和初始目标队列
- 输出内容：
  - 当前所在楼层
  - 目标楼层数量（即外部请求数量）
  - 初始目标队列（从当前楼层开始，按顺序连接所有外部请求楼层）

### 3.8. 内部请求输入模块
- 对应函数：`Input_Internal_Requests(Elevator *Elev)`
- 功能：按外部请求的顺序，接收每个外部请求对应的乘客内部目标楼层
- 输入流程：
  1. 依次提示用户输入每个外部请求楼层的乘客目标楼层
  2. 将输入的内部目标楼层存入电梯结构体的内部请求队列
  3. 自动更新内部请求数量

### 3.9. LOOK算法核心调度模块
- 对应函数：`Look_Sort(Elevator *Elev)`
- 功能：合并外部请求和内部请求，按照LOOK算法规则生成最终的运行路径
- 调度逻辑：
  1. 根据电梯的初始运行方向，分别处理上行和下行两种情况
  2. 遍历所有内部请求，将其添加到对应的Red数组或blue数组中（自动去重）
  3. 对添加了内部请求的Red数组和blue数组重新进行排序
  4. 处理方向翻转后的请求，将反方向的内部请求存入新的辅助数组
  5. 合并Red数组、blue数组和新辅助数组，生成最终的目标楼层队列
  6. 将最终目标楼层队列存入电梯结构体的`finalTargets`字段

### 3.10. 最终运行结果输出模块
- 对应函数：`Output_finalTargets(Elevator *Elev)`
- 功能：输出电梯的最终完整运行路径
- 输出内容：从电梯初始楼层开始，按顺序连接所有最终目标楼层，用箭头"→"表示电梯的运行方向和停靠顺序

## 4. 业务流程设计

### 4.1. 整体业务流程

#### 4.1.1. 系统启动与初始化
1. 打印系统启动标题和分隔线
2. 定义Elevator结构体变量
3. 调用`Init_Elevator`函数初始化电梯状态

#### 4.1.2. 外部请求输入与处理
1. 调用`Input_External_Requests`函数输入并校验外部请求
2. 调用`Determine_Initial_Direction`函数确定初始运行方向
3. 调用`Output_ElevatorInfo`函数输出电梯初始信息

#### 4.1.3. 内部请求输入与调度
1. 调用`Input_Internal_Requests`函数输入内部目标楼层
2. 调用`Look_Sort`函数执行LOOK算法，生成最终运行路径

#### 4.1.4. 结果输出与程序结束
1. 调用`Output_finalTargets`函数输出最终运行结果
2. 程序正常结束
